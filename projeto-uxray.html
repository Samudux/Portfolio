PRODUCT INTELLIGENCE AI PRODUCT DESIGN

UXRay

Product Intelligence Engine

Context

UXRay is a product intelligence engine designed to answer a simple question: why do some products work — and others don't? Instead of analyzing interface at a surface level, the system extracts underlying structures such as mental models, growth mechanisms and product defensibility.

Problem

Most product analysis is:

  • Subjective
  • Visual-focused
  • Not actionable

This makes decision-making inconsistent and unreliable.

Solution

UXRay analyzes digital products and extracts:

  • Mental model
  • Growth model
  • Product moat
  • Strategic signals

The output is structured to support real product decisions.

Product Decisions

Focus on structure over surface

The system prioritizes models and mechanisms instead of UI evaluation.

Decision-oriented output

The analysis is designed to guide action, not just describe.

Layered reading system

Three levels: summary, analysis and technical — adapting depth to the user.

Interface as support layer

UI exists to organize thinking, not to impress visually.

System Decisions

Fixed analysis pipeline

Early analyses showed inconsistent outputs — the same product could generate different structures depending on the response. This made results hard to trust and compare. A fixed pipeline was introduced to enforce a consistent sequence of analysis, reducing variability and stabilizing output.

Separation between generation and interpretation

Raw AI outputs were often verbose, inconsistent or structurally weak. Relying on them directly led to noisy and unreliable results. The system separates generation (AI) from interpretation (engine), allowing structure, meaning and prioritization to be controlled.

Confidence as a core signal

Some outputs appeared convincing even when based on weak signals. This created false certainty and reduced trust in the system. Confidence was introduced as a required signal in every section, making uncertainty explicit instead of hidden.

Expandable graph layer

As analyses accumulated, patterns and relationships started to repeat, but there was no structure to capture them. This limited the system's ability to evolve. A graph layer was designed to store models, patterns and relationships, enabling future intelligence without changing the core pipeline.

Trade-offs

Speed vs. Depth

Initial versions used multiple AI calls to increase depth, but this introduced latency, instability and frequent timeouts. The system was reduced to a single call, prioritizing reliability and response time over maximum depth.

Flexibility vs. Consistency

Allowing flexible outputs created variation in structure, making analyses harder to compare and interpret. A fixed output structure was enforced — flexibility was sacrificed to make the system trustworthy at scale.

Complexity vs. Usability

The internal system includes multiple layers (pipeline, interpretation, graph), which could overwhelm the user if exposed. This complexity was intentionally hidden, allowing the user to interact with a simplified interface while maintaining depth internally.

Design Decisions — Interface as System Layer

The interface was designed as a support layer for the system — its role is to structure and expose the analysis, not to compete with it.

Minimal interface

Early explorations with more visual elements introduced noise and made the output harder to scan. The density of information required a quieter interface. The UI was reduced to the minimum necessary, ensuring the content remains the primary focus.

Layered reading system

Users interacted with the same output in different ways — some needed quick answers, others needed full breakdowns. A single-layer interface forced unnecessary friction. The interface was structured into three levels (summary, analysis, technical), allowing depth without overwhelming the user.

Fixed visual structure

When the layout varied between analyses, users struggled to locate information and build familiarity with the system. This increased cognitive load and reduced efficiency. A consistent visual structure was enforced, making navigation predictable and reducing interpretation effort.

Explicit confidence indicators

Outputs could appear convincing even when based on weak signals. Without a visible indicator, users tended to over-trust the results. Confidence was exposed in the interface to signal uncertainty and guide interpretation.

These decisions are not visual preferences — they are consequences of how the system operates.
